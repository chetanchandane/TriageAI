## Phase 5 Goals: Human-in-the-Loop & System Calibration ŌĆö Sprint 3 + Sprint 6 patch

- [X] **LLM-based safety screening (`agents/safety_agent.py` + `graph/nodes.py`) ŌĆö Sprint 6 (Apr 2026)** ŌĆö Replaced the entire regex-based safety layer with a single context-aware LLM call. The previous two-stage approach (regex patterns ŌåÆ LLM confirmation) still produced false positives because regex patterns like `\bheart\s+attack\b` matched historical mentions (e.g. "I had a heart attack five years ago and just need a refill"). Even when the LLM Stage 2 correctly identified these as non-acute (`triggered_by="rules_only"`), a bug in `synthesis_node` blindly overrode urgency to EMERGENCY for any `is_potential_emergency=True` ŌĆö ignoring the triage agent's context-informed assessment. New logic: a single LLM screening prompt evaluates whether the message describes an ACTIVE, CURRENT, LIFE-THREATENING emergency. The prompt explicitly handles: past medical history ("I had", "history of", "five years ago"), chronic/stable conditions, medication refills that mention serious past conditions, and managed conditions (epilepsy, asthma, chronic pain). `triggered_by` is now either `"llm"` (confirmed emergency, short-circuits graph) or `"none"` (normal triage flow). Also fixed `synthesis_node` to check `state.get("is_emergency")` instead of `safety.get("is_potential_emergency")` ŌĆö only overrides urgency for confirmed emergencies that short-circuited the graph, not for cases the triage agent already assessed. LLM failure still defaults to `False` (avoids false positives from outages). Decision: regex patterns are inherently context-blind ŌĆö they match keywords regardless of tense, negation, or surrounding context. An LLM call costs ~200ms more but eliminates an entire class of false positives that no amount of regex refinement could fix. The synthesis_node fix ensures the triage agent's assessment is respected when the safety screen routes through it.
