## Phase 4 Goals: Agentic Orchestrator & Reasoning Loop — Sprint 2

- [X] **Cyclic agentic graph** — Replaced the linear Safety → Triage pipeline with a cyclic LangGraph workflow. The new graph: `safety_node → (emergency? END : triage_agent_node) → (tool_calls? tool_node → triage_agent_node : synthesis_node) → END`. The triage agent can now loop: call tools, read results, call more tools, and only exit when it has enough context to produce a final assessment. Decision: this is the core architectural shift from "chatbot" to "Clinical Decision Support System" — the AI dynamically pulls from Supabase (history) and ChromaDB (policy) to ground its decisions.
- [X] **State redesign (`graph/state.py`)** — Added `messages` field with LangGraph's `add_messages` reducer so tool outputs append to a running conversation log instead of overwriting the original message. Added `patient_id`, `medical_history`, `policy_context`, `draft_reply`, `is_emergency` (short-circuit flag), and `staff_approved` (for future HITL). Decision: the `messages` list is essential for agentic loops — the LLM needs to "see" its own tool calls and the results to reason about what to do next.
- [X] **LangChain tool wrappers** — Wrapped the 3 MCP tools (`get_patient_history`, `search_hospital_policy`, `get_available_slots`) as LangChain `@tool` functions in `graph/nodes.py`. These are bound to the Gemini model via `.bind_tools()` so the LLM can invoke them during reasoning. Decision: LangChain's tool interface is required for `langgraph.prebuilt.ToolNode` to automatically route tool calls to the correct Python functions.
- [X] **Triage agent node** — Implemented `triage_agent_node` using `ChatGoogleGenerativeAI` (Gemini 2.5 Flash) with all 3 tools bound. The system prompt instructs the agent to: (1) analyze the patient message, (2) call tools as needed for history, policy, or scheduling, (3) produce a final triage assessment. The node returns an AIMessage that either contains tool_calls (routed to tool_node) or a final response (routed to synthesis). Decision: binding tools to the LLM and letting it decide what to call (rather than hardcoding the sequence) enables multi-intent handling — e.g., a single message asking for both a refill and an appointment triggers both `search_hospital_policy` and `get_available_slots`.
- [X] **Tool node** — Used `langgraph.prebuilt.ToolNode` which automatically maps LLM `tool_calls` to the corresponding Python functions and returns ToolMessages. Connected back to `triage_agent_node` to form the reasoning loop. Decision: ToolNode is the standard LangGraph pattern — it handles argument parsing, error wrapping, and message formatting, so we don't need to implement any of that manually.
- [X] **Synthesis node** — Added a `synthesis_node` as the final step before END. It extracts the TriageResult from the agent's conversation using a two-phase strategy: (1) try to parse JSON from the agent's last message, (2) if that fails, make an explicit structured extraction call to Gemini with `response_schema=TriageResult`. Decision: separating synthesis from the reasoning loop is more robust — the tool-calling Gemini model produces natural language, and the synthesis step reliably converts it to the structured schema the staff dashboard expects.
- [X] **Safety short-circuit** — When the safety node detects an emergency (`is_emergency=True`), the graph routes directly to `synthesis_node`, skipping the triage agent entirely. The synthesis node then tags the result with `safety_flagged`, `safety_reason`, and overrides urgency to `EMERGENCY`. Decision: avoids unnecessary LLM latency for emergencies — the safety gate catches it in milliseconds (rule-based) and immediately exits.
- [X] **Routing logic** — Implemented `_route_after_safety` (emergency → synthesis, else → triage_agent) and `_should_continue` (tool_calls → tool_node, else → synthesis) as conditional edge functions. Decision: these two functions define the entire control flow of the agentic loop.
- [X] **Streamlit integration** — Updated `_run_workflow(msg, patient_id)` in `app/streamlit_app.py` to pass the patient's `patient_id` from session state to the workflow. The `run_triage_workflow` function signature now accepts `patient_id` but remains backward-compatible (defaults to empty string). Decision: the agent needs the patient_id to call `get_patient_history`.
- [X] **New dependency** — Added `langchain-google-genai>=2.0.0` to `requirements.txt`. This provides `ChatGoogleGenerativeAI` which supports `.bind_tools()` for Gemini function calling within LangGraph. Decision: needed for the standard LangGraph + Gemini agentic pattern (ToolNode expects LangChain message types).
- [X] **End-to-end tests** — Verified 3 scenarios:
  - Emergency ("sharp chest pain"): Safety gate catches it via rule match → short-circuits → EMERGENCY urgency. No LLM triage call made.
  - Multi-intent ("refill Lisinopril + schedule appointment"): Agent calls `search_hospital_policy` and `get_available_slots` → loops back with results → synthesizes structured TriageResult with intent "Refill", NORMAL urgency, and relevant checklist.
  - Admin ("billing question"): Agent calls `search_hospital_policy` → synthesizes with intent "Billing", queue "Billing".

## Phase 5 Goals: Human-in-the-Loop & System Calibration — Sprint 3 + Sprint 6 patch

- [X] **Two-stage emergency confirmation (`agents/safety_agent.py` + `graph/nodes.py`) — Sprint 6 patch (Mar 2026)** — Redesigned the safety layer from a single-layer screen into a two-stage confirmation system. Previously, either a rule match OR an LLM flag was sufficient to set `is_emergency=True` and short-circuit the entire graph, causing chronic/historical/mild symptoms to be incorrectly escalated (e.g. "seizure last year", "back pain for months", "shortness of breath when climbing stairs"). New logic: Stage 1 rule fires → Stage 2 LLM confirmation with a focused "is this ACTIVE and ACUTE?" prompt. Only when BOTH agree (`triggered_by="rules+llm"`) does the graph short-circuit. Single-layer flags (`rules_only`, `llm_only`) set `is_potential_emergency=True` but `is_emergency=False`, routing through the triage agent which uses patient history tools to make a context-informed urgency call. Also fixed critical bug: LLM API failures previously defaulted to `is_potential_emergency=True` (outages created false positives). Now defaults to `False` — the rule-based layer already ran, so an LLM outage should not override a clean rule scan. Decision: this implements the professor's "multiple stages + tool-assisted confirmation" recommendation while maintaining 0% false negatives (all suspicious cases are still flagged and elevated, just not blindly short-circuited). The triage agent seeing a `safety_result.is_potential_emergency=True` flag will naturally weight urgency higher when assessing.
- [X] **Safety calibration (`agents/safety_agent.py`)** — Refined `EMERGENCY_PATTERNS` to require crisis modifiers, fixing the "false emergency" over-flagging. Previously broad patterns like `\bchest\s+pain\b` would flag routine mentions (e.g., "I sometimes have chest pain when I exercise"). Now requires modifiers: `\b(sudden|crushing|severe|sharp|intense)\s+chest\s+pain\b`. Similarly, `\bheart\s+racing\b` → `\bheart\s+(is\s+)?racing\s+with\s+(dizziness|fainting|...)\b`. Also removed standalone `\b(confusion|slurred\s+speech)\b` and `\bvision\s+(loss|blurred|double)\b` — these now require `\bsudden\s+` prefix. Decision: 0% false negatives is still the target, but the old patterns had too many false positives (flagging refill requests that mention "pain"). Crisis modifiers ensure only acute presentations are flagged by rules; the LLM fallback still catches edge cases.
- [X] **LLM prompt negative constraints** — Added a `NEGATIVE CONSTRAINTS` section to the `_llm_safety_screen` prompt: "Do NOT flag routine refills, scheduling, billing, chronic/stable symptoms lasting >1 week, or mild symptoms without urgency modifiers." Decision: the LLM was too conservative — flagging anything that mentioned a body part or symptom. Explicit negative examples give the model a calibrated reference.
- [X] **State extension (`graph/state.py`)** — Added `patient_email: Optional[str]` (needed for the communication node to send emails) and `hitl_status: Optional[str]` (tracks workflow state: `"pending_review"`, `"approved"`, `"auto_completed"`, `"dismissed"`). Decision: these fields are essential for the HITL flow — the Streamlit app uses `hitl_status` to determine whether to show a message in the Pending Approvals tab.
- [X] **Draft reply node (`graph/nodes.py`)** — New `draft_reply_node` uses the policy agent's RAG (`get_relevant_policy` + `generate_draft_reply`) to produce a policy-grounded draft reply for every triaged message. Runs after synthesis, before any communication. Decision: separating draft generation from communication enables staff to review and edit the draft before it's sent.
- [X] **Communication node (`graph/nodes.py`)** — New `communication_node` calls `send_resolution_email` with the (potentially edited) draft reply. Sets `staff_approved=True` and `hitl_status="approved"`. This is the node that gets interrupted by the HITL checkpointer. Decision: making communication a discrete graph node (rather than inline Streamlit logic) means the graph can pause here and the UI is decoupled from the action.
- [X] **MemorySaver checkpointer (`graph/workflow.py`)** — Integrated `langgraph.checkpoint.memory.MemorySaver` as the persistence layer. Every node's output is checkpointed against a `thread_id`, enabling state recovery and mid-workflow pauses. Decision: `MemorySaver` is the right choice for this milestone — zero setup, no external database. In production, this would be swapped for `SqliteSaver` or a Redis-backed checkpointer.
- [X] **Conditional interrupt strategy** — Used `interrupt_before=["communication_node"]` with two separate end-path nodes: `communication_node` (interrupted, for NORMAL/HIGH/EMERGENCY) and `auto_communicate` (not interrupted, for LOW). The routing function `_route_after_draft` checks `triage_result.urgency` to decide the path. Decision: LangGraph's `interrupt_before` is statically defined at compile time — it always interrupts the named node. To allow LOW-urgency messages to flow through without staff review, we route them to a separate `auto_communicate` node that is not in the interrupt list. Both nodes call the same `send_resolution_email` function.
- [X] **Thread ID support** — `run_triage_workflow` now accepts an optional `thread_id` (auto-generates a UUID if not provided). The thread_id is embedded into `triage_result` so the Streamlit app can later resume the workflow. Decision: thread_id is the key for state recovery — each patient message gets a unique thread, and the staff dashboard uses it to call `get_workflow_state` and `resume_workflow`.
- [X] **Resume workflow** — New `resume_workflow(thread_id, edited_draft)` function in `graph/workflow.py`. If staff edited the draft, it calls `app.update_state(config, {"draft_reply": edited_draft})` before resuming with `app.invoke(None, config)`. Decision: this is the standard LangGraph HITL pattern — update state, then invoke with `None` to continue from the interrupt point.
- [X] **State inspection** — New `get_workflow_state(thread_id)` function returns the full state dict for a given thread. Used by the staff dashboard to display the interrupted workflow's triage result, draft reply, and safety flags. Decision: exposes the checkpointer's state without requiring the staff to understand LangGraph internals.
- [X] **Streamlit HITL dashboard** — Added a third tab **"Pending Approvals"** to the Streamlit app. It lists all messages where `triage_result.hitl_status == "pending_review"`, showing the AI analysis, an editable `st.text_area` for the draft reply, and "Approve & Send" / "Dismiss" buttons. Approve calls `resume_workflow` and updates the message store. Dismiss marks the message as dismissed. Decision: three tabs (Patient / Staff / Approvals) keeps the workflow clear — patients submit, staff reviews the queue, and the approvals tab is specifically for HITL intervention.
- [X] **Patient email pass-through** — Updated `_run_workflow` in `app/streamlit_app.py` to pass `patient_email=patient.email` through to the workflow. The communication node uses this to send emails. Decision: the patient's email was already in session state from login; now it flows through the entire graph.
- [X] **HITL status badges** — Staff queue now shows status badges: ⏸️ Pending, ✅ Sent, ⚡ Auto for each message based on `hitl_status`. Patient view hides internal fields (`thread_id`, `hitl_status`, `draft_reply`) from the triage summary.
- [X] **End-to-end tests** — Verified 4 scenarios:
  - Safety calibration: "chest pain when exercising" → NOT flagged by rules (PASS). "Sudden crushing chest pain" → flagged (PASS). "Heart racing alone" → NOT flagged (PASS). "Heart racing with dizziness" → flagged (PASS).
  - Emergency HITL: "Severe crushing chest pain" → safety flags → synthesis → draft_reply → **interrupted** before communication_node. `get_workflow_state` retrieves draft. `resume_workflow` with edited draft → email sent → `hitl_status = "approved"`.
  - Routing logic: LOW → `auto_communicate` (no interrupt). NORMAL/HIGH/EMERGENCY → `communication_node` (interrupted).
  - Sprint 1 regression: All 8 original MCP tool tests still pass.

This means staff only receive a case once the AI is fully satisfied — every clarification has already happened in-chat. Decision: the agent, not the gate, determines when the conversation is complete. This is architecturally correct because the agent knows what information it needs to produce a high-confidence triage result.
